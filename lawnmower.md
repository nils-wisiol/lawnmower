## 1. Overview / Elevator Pitch
- One-paragraph summary: mow every tile of a garden exactly once, routing
  around obstacles. Puzzle game about efficient coverage.
- Genre, tone, target platform (browser), target audience.
- The single "aha": optimal mowing is a hard routing problem (see §6).

## 2. Gameplay
- **The grid**: garden as a 2-D grid of tiles. A tile has two **orthogonal**
  traits (not a single enum): **passable** (can the mower enter it?) and
  **mowable** (does it count toward the objective / does re-entry fail?). See
  §5 for why this split matters for forward-compat.
  - v1 uses just two combinations: *grass* (passable + mowable) and *obstacle*
    (impassable). The **passable-but-not-mowable** tile (e.g. a garden path you
    may cross freely) is a valid third combination we design *for* now but do
    not ship in v1.
- **Obstacles**: lakes, flowers, trees, etc. — impassable tiles the player
  routes around.
- **The mower / player**: how it occupies a tile.
- **Starting position**: **fixed per level** for now (defined by the level /
  generator). *Later levels may vary this (player-chosen, multiple valid
  starts) — parked in §9.*
- **Movement rules**: **4-way only** (N/S/E/W), one tile per step. The mower
  may only enter **passable** tiles. Re-entering a **mowable** tile is a fail
  (see below) — the rule is scoped to *mowable* tiles, not "any visited tile,"
  so a future passable-non-mowable tile can be crossed more than once without
  changing the rule. *Future variants: 8-way movement, or a hexagonal world —
  noted in §9 roadmap, out of scope for v1.*
- **Win condition**: every mowable tile mowed.
- **Fail condition**: the re-enter-a-mowable-tile fail from Movement rules is a
  **hard fail** (immediate loss / level restart). No partial-credit revisits in
  v1. (Exceeding a level's optional time limit, if set, also fails — see
  scoring.)
- **No undo**: moves are committed on input; the only recovery from a mistake
  is restarting the level. Preserves the speed-vs-caution tension (scoring).
- **Scoring & efficiency metric**: primary score is **wall-clock completion
  time** — fastest full mow wins. Levels *may* impose a **time limit**;
  exceeding it is a fail. (Note: revisits are already a hard fail, so every
  valid completion is a perfect mow; speed is the differentiator.)
  - **Timer start**: a **per-level config option**, defaulting to **start on
    first move** (planning before the first move is free). Levels may override
    to start on load.
  - **Focus loss**: the timer **keeps running** if the tab is backgrounded or
    loses focus — no pause. (Consistency with wall-clock scoring; players learn
    not to leave mid-run.)
  - Open: is speed purely execution, or do we reward plan quality too? The
    tension is "commit to a route quickly vs. risk a dead-end fail." See §10.
- **Level source & progression**: levels are produced by a **seedable random
  generator** — same seed reproduces the same level (shareable / repeatable
  runs, daily challenge, deterministic tests). Difficulty scales via seed
  parameters (grid size, coverage/obstacle density). Generation method in §5.
  *Later versions may ship a curated/handcrafted level set alongside.*
- **Controls**: **arrow keys** (desktop) and **swipe gestures** (touch/phone),
  mapped to the four cardinal directions.
- **Movement cadence**: mower advances **one tile per key press / swipe**
  (discrete, input-driven — not continuous auto-motion). Wall-clock scoring
  still applies, so the pressure is on thinking + inputting fast.
- **Game modes**: candidates — puzzle/campaign, endless/random, daily
  challenge (fixed seed per day), sandbox. Per-mode scope for v1 vs later: §9.

## 3. Art & Visual Style
- **Direction / mood**: **pixel art in the style of Monkey Island 2** as the
  *aspirational target* — rich, detailed, painterly SCUMM-era pixel work. Must
  read clearly as a **garden** (grass, water, flowerbeds, trees, hedges), not an
  abstract puzzle board.
- **Art source (for now): AI-generated by Claude, authored as code/data.** No
  image-generation model is available in-session, so v1 art is **programmatic
  pixel art** — sprites defined as indexed-palette pixel matrices / hand-written
  canvas-draw routines, bundled inline (fits the self-contained static-hosting
  constraint). **Honest expectation:** this yields *readable, characterful*
  pixel art trending toward the MI2 look, **not** true hand-painted MI2 fidelity.
  The theme layer (below) is what makes this acceptable — art can be upgraded
  later (real artist, or a future image model) with **no logic changes**.
  "For now" is deliberate: better asset sourcing is expected eventually.
- **Tile & entity visuals**: unmowed vs mowed grass, obstacle types (lake,
  flower, tree), the mower sprite, start marker, mow trail.
- **Feedback visuals**: legal-move affordances, imminent dead-end warning,
  win/fail states, timer display.
- **Theming (v1 single theme, built to reskin)**: all sprites/colors are pulled
  through a **theme layer**, so a full **retheming** to a different setting/art
  style is an asset+data swap, not a code change. v1 ships one garden theme;
  retheming is on the roadmap (§9) and the implementation is prepared for it
  (see §5 theming layer). Light/dark is a theming concern too.
- **Asset pipeline & constraints**: sprites authored as **code/data** (indexed-
  palette pixel matrices or canvas-draw routines) on a fixed pixel grid; integer
  scaling for crispness; embedded/bundled inline (no external binary art files),
  keeping the app self-contained for static hosting; defined animation budget.
  A shared palette definition supports the theming layer and future colorblind
  palettes (§9).
- **Geometry-awareness**: art must survive the square→hex generalization (§5) —
  tiles addressed via the Board layout, not hardcoded square coordinates.

## 4. UX & Player Experience
- **First-time experience**: onboarding / how rules are taught (tutorial level,
  no wall of text).
- **Core loop feel**: start → plan → execute → win/fail → retry/next, kept
  fast and low-friction (instant restart, next-level flow).
- **Readability**: grid must be legible at a glance; clear current position,
  what's mowed, where obstacles are.
- **Feedback & juice**: input responsiveness, move/mow feedback, fail clarity
  ("you revisited *here*"), timer prominence without anxiety overload.
- **Mobile vs desktop**: swipe ergonomics, screen-size adaptation, one-handed
  play; no reliance on hover.
- **Accessibility**: keyboard play is inherent in v1 (arrow keys) and no
  hover-only affordances. **Deferred to roadmap (§9):** colorblind palettes,
  reduced-motion, and touch-target sizing.
- **Meta UX**: seed entry/sharing, level select, best-time display, settings.

## 5. Implementation (Browser)
- **Tech stack** (recommendations — see chat for rationale, refine here):
  - Language: **TypeScript**.
  - Build/dev: **Vite** (fast dev server, static build → GitHub Pages).
  - Rendering: **Canvas 2D** behind a render interface (generalizes to hex /
    arbitrary geometry far better than CSS-grid/DOM; WebGL swap possible later).
  - UI/menus: keep core **framework-free**; optionally a lightweight lib
    (Preact/Svelte) only for menu chrome. Decide during §9 slice work.
  - Tests: **Vitest** (unit) + **Playwright** (end-to-end).
  - Lint/format: **ESLint + Prettier**.
- **Architecture (must generalize for the roadmap)**:
  - Central abstraction: a **Board/Topology interface** — cells, a neighbor
    function, a direction set, cell→pixel layout, and input-direction mapping.
    Square grid and hex grid become two implementations of one interface.
  - **Core game logic operates on the abstract cell graph** (mow state, move
    validation, win/fail, timing) — geometry-agnostic, so 8-way and hex fall
    out of new Board impls, not rewrites.
  - **Generator also targets the abstract graph** (random self-avoiding walk
    works on any adjacency structure).
  - **Teleportation is already expressible** (making it explicit): a teleport is
    just a **non-geometric adjacency edge** in the Board's neighbor function.
    Because movement/win/fail operate on the abstract cell graph, teleport tiles
    need **no core-rule changes** — a teleport feature (roadmap §9) only adds
    input-direction mapping and rendering, not new game logic. Keep the neighbor
    function free to return non-adjacent cells; don't assume neighbors are
    geometrically adjacent.
  - **Theming layer**: rendering pulls all sprites/colors from a **swappable
    theme**, so retheming (§3, roadmap §9) is an asset/data swap, not a code
    change. v1 ships one garden theme; keep no theme constants hardcoded in
    render/logic.
  - **Cell model is trait-based, not enum-based** (forward-compat guardrail):
    each cell carries orthogonal `passable` and `mowable` flags (see §2), *not*
    a single `grass | obstacle` type. Core rules read these traits — movement
    checks `passable`, the win condition counts unmowed `mowable` cells, the
    revisit fail fires only on `mowable` cells. This means
    **passable-but-not-mowable** (and other trait combinations) drop in as
    data, with **no change to the movement/win/fail logic**. The generator's
    "gaps → obstacles" step just sets traits, so it stays compatible too.
  - Clean separation: model / rendering / input, each swappable and testable.
  - **Injected clock**: the timer/scoring reads from an injected time source,
    never `Date.now()` directly. Keeps wall-clock scoring deterministically
    testable (a fake clock in unit tests) despite time being non-deterministic.
  - **Input pipeline**: one move per discrete input. **No key-repeat** — holding
    a direction key does *not* auto-advance; each move needs a distinct press
    (or swipe). Whether inputs queue vs. coalesce during a move animation is a
    minor M2 implementation detail (default: ignore input until the move
    settles), not a design blocker.
- **Data model & storage**: **no server backend** — app ships as **static
  files**; all persistence is **in-browser** (localStorage/IndexedDB) for best
  times, settings, seed history.
- **Level formats** (two):
  1. **Short form** — a **generator-version tag** + seed + a few config options
     (size, difficulty), for RNG-generated levels. Compact, shareable string.
     The version tag pins which generator algorithm produced the level, so a
     future algorithm change doesn't silently make an old seed decode to a
     *different* level (which would break shared seeds, daily challenges, and
     stored best-times). **Policy: keep the number of live generator versions
     minimal.** We are *not* committed to carrying old generator code forever —
     prefer rewriting/regenerating tests over maintaining stale algorithms. The
     version tag exists so mismatches are *detected* (and can be handled
     explicitly), not so every version is preserved indefinitely.
  2. **Long form** — an explicit spelling-out of the map (cells with their
     `passable`/`mowable` traits, start) plus config, for levels *not* from the
     seeded RNG (handcrafted / edited). Long form is the canonical superset;
     short form expands (via its pinned generator version) into it.
- **Testing strategy**: deterministic tests via fixed seeds; unit tests on
  model + generator; Playwright e2e for input→win/fail flows.
- **Project layout**: directory structure sketch.

## 6. Why This Is Not Trivial to Solve
- Frame the optimal-mow problem formally: cover-every-tile-once path on a grid
  graph with holes (obstacles).
- Relation to a **Hamiltonian path** on the grid graph → NP-completeness of
  Hamiltonicity in grid graphs with holes.
- Distinguish: (a) does a perfect mow exist? (b) minimum-revisit mow if not.
- **Solvability by construction**: because we generate a level *from* a random
  self-avoiding walk (§5) and turn only the uncovered cells into obstacles, the
  walk itself is a guaranteed perfect mow — we never face the NP-hard
  *decide-if-solvable* problem at generation time. The player, however, still
  faces the hard *find-a*-perfect-mow problem. Spell out this asymmetry.
- Note honestly where the analogy is exact vs. hand-wavy (to firm up later).

## 9. Roadmap / Milestones

### Path to v1 (ordered milestones)
Each milestone is a shippable, testable slice; later ones build on earlier.

- **M0 — Project scaffold.** `git init`, Vite + TypeScript, ESLint/Prettier,
  Vitest + Playwright wired up, empty GitHub Pages CI deploy green. *Done when:*
  a blank page deploys from CI and `test` runs.
- **M1 — Core model.** Board/Topology interface + square-grid impl;
  **trait-based cell model** (`passable`/`mowable`, §5) rather than a
  grass/obstacle enum; move validation (4-way, enter-only-passable, hard-fail
  on re-entering a mowable tile); win/fail detection. Pure logic, no rendering.
  *Done when:* unit tests drive a hardcoded level to win and to fail — and a
  test proves a passable-non-mowable tile can be crossed twice without failing,
  locking in the forward-compat guarantee.
- **M2 — Playable render + input.** Canvas rendering of a hardcoded level;
  arrow-key input; visible mower, mowed trail, win/fail state. *Done when:* a
  human can play one level to completion in the browser.
- **M3 — Generator.** Seeded self-avoiding-walk generator with coverage floor,
  gaps→obstacles; fixed start from the walk. Short-form seed decoding.
  *Done when:* a seed yields a solvable level, reproducibly (deterministic test).
- **M4 — Scoring & timing.** Wall-clock timer, optional per-level time limit,
  win screen with time, instant restart / next-level flow. *Done when:* a full
  play loop (start→win→next) works with recorded time.
- **M5 — Persistence & seed sharing.** In-browser storage (best times, seed
  history); enter/share a seed via a **URL hash** that carries the level's
  short-form code and updates as levels load, so copying the URL shares the exact
  level. *Long-form level load is **deferred** past M5* — the seeded short-form
  path covers all v1 levels, and hand-authored maps already load via the ascii
  helper for tests; full Level↔string long-form serialization lands only when a
  handcrafted level pack needs it (roadmap, §9 "Content"). *Done when:* best time
  survives reload; a shared seed reproduces a level.
- **M6 — UX & art polish.** Onboarding/tutorial level; **Claude-authored
  programmatic pixel-art garden theme** (§3) replacing M2 placeholders via the
  theme layer; mobile swipe input; feedback juice. *Done when:* a first-time
  player understands the rules unaided on mobile+desktop, and the garden theme
  is in place. *(Colorblind palettes, reduced-motion, and touch-target sizing
  are roadmap items, §9 — not required for v1.)*
- **v1 release** = M0–M6 deployed to GitHub Pages.

### Out of scope for v1
- Multiple game modes beyond single-level generated play (daily/campaign later).
- Handcrafted level packs, leaderboards/online features.

### Future variants & features (post-v1)
- **World/rules**: 8-way movement; **hexagonal grid world**;
  **teleportation tiles** (architecture-ready — non-geometric graph edges, §5);
  variable/player-chosen start.
- **Content**: curated/handcrafted level packs.
- **Presentation**: **retheming / alternate art themes** (theming layer already
  in place, §3/§5); **audio & sound feedback**; WebGL renderer.
- **Accessibility**: colorblind palettes; reduced-motion; touch-target sizing.

## 10. Open Questions
Running list of undecided design points to resolve across iterations.
- **Scoring depth**: is speed purely execution, or do we also reward plan
  quality? (From §2.)
- **Variable start position**: if/when levels allow player-chosen or multiple
  valid starts, and how that interacts with solvability. (From §2.)
- **Hex + swipe input**: 6 neighbors vs. a swipe's ~4–8 rough directions — how
  to map gestures on hex. (Deferred, but capture now. From §5/§9.)
- **Time-limit sourcing**: how a level's optional time limit is derived
  (fixed? scaled to grid size / optimal path length?). (From §2.)
- **UI framework**: framework-free vs. Preact/Svelte for menus — decide at the
  slice that first needs real menu chrome. (From §5.)
- **Obstacle / tile semantics**: the *architecture* is settled — trait-based
  cells (`passable`/`mowable`) make passable-non-mowable and other combinations
  data, not rework (§5). Still open: the *gameplay* details — which extra tile
  kinds we actually introduce, when, and how the generator places them. (§2.)
